This document outlines the expected behaviours for all developers. These practices should be referenced and followed in all development work.

## Core Development Principles

### 1. **Explain Before Implementing**
- **Always provide a non-technical summary** of what you plan to implement before coding
- Use **bullet points** to clearly outline the concept and approach
- **Wait for approval** before proceeding with implementation
- Explain the "why" behind the approach, not just the "what"

### 2. **Clarify Ambiguity**
- **Ask questions** when requirements or expectations are unclear
- **Don't make assumptions** about what is wanted
- Seek clarification on scope, approach, or technical details when uncertain
- Better to ask and get it right than assume and get it wrong

### 3. **Stay Within Scope**
- **Only make changes you have been explicitly asked to make**
- Do not add "nice to have" features or improvements unless requested (but these should be suggested)
- Resist the urge to refactor or "improve" code that wasn't part of the task (again, this can be suggested as a follow on activity)
- Focus on the specific problem being solved

### 4. **Preserve Naming Conventions**
- **Do not alter variable names arbitrarily**
- Maintain consistency with existing naming patterns
- Only change names when explicitly requested or when they are clearly incorrect
- Respect the established codebase conventions

### 5. **Challenge Poor Decisions**
- **Speak up when something seems problematic**
- If a suggestion or request seems ill-advised, say so and explain why
- Offer alternative approaches when challenging existing decisions
- Technical expertise should guide decision-making, not just following orders

### 6. **Validate Against Project Standards**
- **Always cross-check your approach** against project documentation
- Ask: *"Does this align with the defined approach set out for the project?"*
- Reference architecture principles, patterns, and established practices
- If your approach doesn't align, redesign rather than proceed
- Specifically check against:
  - Architecture Principles documentation
  - Service Patterns documentation  
  - Feature Patterns documentation
  - Existing codebase patterns

### 7. **Holistic Consideration**
- **Consider all above principles** in every piece of work
- Don't focus on just one aspect while ignoring others
- Take a step back and evaluate the complete approach
- Ensure consistency across all aspects of the solution



## Working with This Document

- **Reference these practices** in all development discussions
- **Call out violations** when they occur
- **Update this document** when new practices are identified
- **Use this as a checklist** before submitting any work

## Success Metrics

Good development work on this project:
- ✅ Follows established patterns and principles
- ✅ Solves the specific problem without scope creep
- ✅ Integrates cleanly with existing infrastructure
- ✅ Can be easily maintained and extended
- ✅ Is properly documented and explained

Poor development work:
- ❌ Makes assumptions without clarification
- ❌ Adds unnecessary complexity or features
- ❌ Breaks established patterns
- ❌ Creates unused or dead code

- ❌ Ignores project standards and conventions
